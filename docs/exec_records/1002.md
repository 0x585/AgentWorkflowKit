## 变更说明

- 将 `workflow_guard` 改成“先执行、后检查”的顺序：入口成功时不做中央版本核对。
- 当入口失败时，`workflow_guard` 才调用 `check_release.py`；如果状态是 `outdated`，则自动执行 `apply_release.py` 并重试一次入口。
- 如果入口失败且当前 release 其实是最新的，则直接返回原始失败码；如果同时检测到 `drift` / `invalid`，则给出版本校验失败提示。
- 调整 `workflow_kit_lib`，使 worktree 发版时优先读取 `.workflow-kit/source.json` 的 `repo_id`，并把渲染后的 `workflow_repo_root` 固定回中央主仓库根路径，避免把 worktree 临时路径写进下游元数据。
- 为中央仓库补齐 `agent_workflow_kit` 包、`PublicWorkRegisterService`、空白 `pending-worklist.md`，并让 `public_work_register_sync.py` / `claim.py` 在运行时自动创建缺失目录。
- 更新 README / AGENTS / USAGE 与模板文案，明确新行为是“失败后才核对并升级”。
- 发布 `full_codex_flow@1.0.3`。
- 新增两条 workflow release 回归测试，以及一组 `PublicWorkRegisterService` 基础测试。

## 风险与回滚

- 风险：如果入口本身失败原因与 workflow 版本无关，新的 guard 会在失败后额外做一次 release 检查；不过在 `current` 状态下会直接返回原始失败码，不会自动改动仓库。
- 风险：本次引入了中央仓库自己的 `public_work_register` service 骨架，后续若希望与 `AgentTask` 继续共享实现，需要再评估是否抽成统一模板或库。
- 风险：下游 release 真正应用要等本次分支提交并合入 `main` 后再执行，避免下游指向中央主仓库时读到旧 release。
- 回滚：回退 `workflow_guard` / `workflow_kit_lib` / `public_work_register_*` / `src/main/python/agent_workflow_kit/**` 与 `1.0.3` release 文件，再重新发布上一个稳定 release。
