将中央 workflow 的代码任务顺序明确并固化为：

- 测试
- 审查
- commit
- push / auto-release
- apply 到所有子项目

本次不仅更新文档，还把关键顺序落到受管 hooks / runtime 上，避免出现“规则写了但实际不会拦”的状态。

## 变更说明

- 将代码任务的默认顺序固定为“测试 -> 审查 -> commit -> push / auto-release -> downstream apply”
- 在 `commit-msg` 中新增代码提交守卫：
  - 代码改动提交前，exec record 必须完成 `验证结果` / `审查结果`
  - docs-only 提交不要求通过这条代码任务校验
- 将 downstream apply 从 `post-commit` 挪到 `session_push_autorelease.sh`
  - 先完成默认分支 merge / push
  - 再执行 `python3 scripts/apply_downstreams.py`
- 保留 downstream local commit 的既有自动化语义
  - 下游 fan-out 生成的本地提交使用 `--no-verify`
  - 同时补齐下游自动记录里的 `审查结果` 字段，避免结构落后于新模板
- 更新中央仓库与下游受管文档，明确新顺序和 hook 行为
- 导出模板并发布 `full_codex_flow@1.0.18`

## 审查结果

- 复核了提交流程拦截点，确认把“测试 / 审查”校验放在 `commit-msg`，可以直接利用 commit message 里的 `exec_id` 对应 exec record，而不需要猜测当前活动记录
- 复核了 auto-release 顺序，确认 downstream apply 必须放在 `session_push_autorelease.sh` 内、且在清理 source worktree 之前执行，否则会出现“顺序正确但脚本已把当前 worktree 删掉”的风险
- 复核了下游自动 fan-out 路径，确认新的人工提交守卫不应拦截机器生成的本地提交，因此在 child worktree 提交处显式使用 `--no-verify`

## 风险与回滚

- 风险：
  - 这次把 downstream apply 的触发时点后移到了 auto-release 之后，若后续有人依赖“commit 一落地就先 fan-out”这一旧顺序，需要同步调整操作习惯
  - `commit-msg` 现在会更严格地阻断代码提交；如果某些自动化路径误走人工 hooks，需要显式补 `--no-verify`
- 回滚：
  - 回退 `commit-msg`、`post-commit`、`session_push_autorelease.sh` 与相关模板 / 文档改动
  - 重新发布一个恢复旧顺序的 workflow 版本，并再次 apply 到下游
